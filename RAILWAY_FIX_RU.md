# Исправление ошибки 500 на Railway

## Что было исправлено

### Основная проблема
Ошибка 500 возникала потому, что приложение пыталось запустить одновременно и gunicorn, и встроенный сервер Flask. Когда Railway загружал приложение через gunicorn, код из блока `if __name__ == '__main__':` всё равно выполнялся, что вызывало конфликт.

## Внесенные изменения

### 1. **app.py** - Основные исправления

#### Разделение инициализации базы данных
- Вынесли код инициализации БД в отдельную функцию `initialize_app()`
- Теперь функция вызывается при импорте модуля (для gunicorn)
- Добавлена обработка ошибок с подробным логированием

#### Настройка PostgreSQL для Railway
```python
# Автоматическое добавление SSL режима для PostgreSQL
if database_url.startswith('postgres://'):
    database_url = database_url.replace('postgres://', 'postgresql://', 1)
    database_url += '?sslmode=require'
```

#### Обработчики ошибок
Добавлены error handlers для лучшей отладки:
- `@app.errorhandler(404)` - страница не найдена
- `@app.errorhandler(500)` - внутренняя ошибка сервера
- `@app.errorhandler(Exception)` - все необработанные исключения

### 2. **Procfile** - Исправлен формат запуска

**Было:**
```
web: gunicorn app:app --timeout 120
```

**Стало:**
```
web: gunicorn app:app --bind 0.0.0.0:$PORT --timeout 120 --workers 4
```

Добавлено:
- Явное указание порта через `$PORT`
- 4 рабочих процесса для продакшена

### 3. **railway.toml** - Обновлена команда запуска

**Было:**
```toml
startCommand = "python app.py"
```

**Стало:**
```toml
startCommand = "gunicorn app:app --bind 0.0.0.0:$PORT --timeout 120 --workers 4"
```

### 4. **nixpacks.toml** - Новый файл

Создан новый файл для лучшего контроля сборки:
```toml
[phases.setup]
nixPkgs = ["python311"]

[phases.install]
cmds = ["pip install -r requirements.txt"]

[start]
cmd = "gunicorn app:app --bind 0.0.0.0:$PORT --timeout 120 --workers 4"
```

### 5. **Новые файлы**

- `.env.example` - пример переменных окружения
- `RAILWAY_FIX.md` - подробная инструкция на английском
- `RAILWAY_FIX_RU.md` - этот файл

## Решение проблем

### Всё ещё ошибка 500?

#### Проверьте логи Railway
1. Зайдите в панель Railway
2. Кликните на ваш деплой
3. Нажмите "View Logs"
4. Ищите сообщения об ошибках

#### Частые проблемы:

**Проблема 1: Не установлена DATABASE_URL**
- Убедитесь, что добавили базу данных PostgreSQL
- Проверьте наличие переменной `DATABASE_URL` в настройках

**Проблема 2: Ошибка привязки порта**
- Railway автоматически устанавливает `$PORT`
- Не нужно устанавливать PORT вручную, если нет проблем

**Проблема 3: Таблицы БД не созданы**
- Приложение теперь автоматически инициализируется при старте
- Ищите в логах "Error during app initialization"

**Проблема 4: Ошибки SSL подключения**
- Код автоматически добавляет `?sslmode=require`
- Если проблемы остались, попробуйте в переменных Railway:
  ```
  DATABASE_URL=postgresql://user:pass@host:port/dbname?sslmode=require
  ```

### Ручная инициализация БД

Если нужно вручную инициализировать базу данных:

1. В Railway перейдите в Settings → Deploy Command
2. Временно измените команду на:
   ```
   python init_db.py && gunicorn app:app --bind 0.0.0.0:$PORT --timeout 120 --workers 4
   ```
3. Перезадеплойте
4. Верните команду обратно (только gunicorn)

## Технические детали

### Почему это работает?

**До исправления:**
```
Gunicorn загружает app.py → Выполняется if __name__ == '__main__' → 
Пытается запустить app.run() → КОНФЛИКТ с gunicorn → Ошибка 500
```

**После исправления:**
```
Gunicorn загружает app.py → Выполняется initialize_app() → 
Создаются таблицы БД → Gunicorn работает нормально → Сайт работает
```

### Обработка PostgreSQL

Railway использует PostgreSQL с обязательным SSL. Код автоматически:
1. Преобразует `postgres://` в `postgresql://`
2. Добавляет параметр `?sslmode=require`
3. Это обеспечивает правильное подключение к БД
